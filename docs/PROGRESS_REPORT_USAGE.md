## レポートの概要

### 配信スケジュール
- **配信日時**: 毎週月曜日 9:00（日本時間）
- **配信期間**: プロジェクト期間中（2026年1月6日〜3月30日）
- **配信先**: GitHub Secretsで設定したメールアドレス

### レポートの目的
1. **進捗の可視化**: 予定と実績を比較して遅延を早期発見
2. **透明性の確保**: ステークホルダー全員が同じ情報を共有
3. **アクションの促進**: 遅延時の迅速な対応を可能に

## レポートの見方

### レポートの構成

週次進捗レポートは以下のセクションで構成されています：

```
📊 週次進捗レポート - Week N
├── プロジェクト基本情報
├── 進捗率サマリー
├── 今週完了したタスク
├── 進行中のタスク
└── 来週予定のタスク
```

### 1. プロジェクト基本情報

| 項目 | 説明 | 例 |
|------|------|-----|
| プロジェクト | プロジェクト名 | 顧問ミドル運用ダッシュボード移行 |
| 期間 | 対象週の期間 | 2026-01-06 〜 2026-01-10 |

### 2. 進捗率サマリー

| 項目 | 説明 | 見方 |
|------|------|------|
| **予定進捗率** | 計画上、この週の終わりまでに達成すべき進捗率 | Week 1の終わりなら9%、Week 2の終わりなら22% |
| **実績進捗率** | 現在完了しているタスクのWeight合計 | 完了タスクが少ないと低くなる |
| **差分** | 実績 - 予定の差 | +5%なら予定より進んでいる、-5%なら遅れている |
| **ステータス** | 進捗状況の判定 | 🟢 予定通り / 🟡 やや遅延 / 🔴 要注意 |

#### ステータスの意味

| ステータス | 条件 | 意味 |
|-----------|------|------|
| 🟢 **予定通り** | 実績 ≥ 予定 | 計画通り進んでいます。このまま継続してください。 |
| 🟡 **やや遅延** | 予定 - 5% ≤ 実績 < 予定 | 若干の遅れがあります。リカバリー可能な範囲です。 |
| 🔴 **要注意** | 実績 < 予定 - 5% | 大きな遅延が発生しています。対策が必要です。 |

### 3. 今週完了したタスク

**表示内容**:
- タスクID（例: TASK-001）
- タスク名
- Weight（進捗率への貢献度）

**見方**:
```
✅ 今週完了したタスク (2件)
  TASK-001  既存BigQuery DWHテーブルの整合性確認  Weight: 2
  TASK-004  Looker/Looker Studio環境のセットアップ  Weight: 2
```

- **タスク数**: 多いほど順調に進んでいる
- **Weight合計**: 大きいほど進捗への貢献が大きい
- **0件の場合**: 今週完了したタスクがない（要注意）

### 4. 進行中のタスク

**表示内容**:
- 現在作業中のタスク一覧
- ステータスが "in_progress" のタスク

**見方**:
```
🔄 進行中のタスク (1件)
  TASK-002  スプレッドシート48項目とBigQueryマッピング分析  Weight: 5
```

- **1-2件が適切**: 集中して作業できている
- **3件以上**: タスクが分散している可能性（要確認）
- **0件**: 次のタスクに着手していない（要注意）

### 5. 来週予定のタスク

**表示内容**:
- 次週に予定されているタスク
- スケジュールに基づく予定

**見方**:
```
📅 来週予定のタスク (3件)
  TASK-005  ダッシュボード要件の体系化  Weight: 4
  TASK-003  データ整合性検証スクリプト作成  Weight: 4
  TASK-006  ダッシュボードワイヤーフレーム作成  Weight: 5
```

- **事前準備**: 必要なリソースや情報を確認
- **依存関係**: 前週のタスクが完了しているか確認
- **工数確認**: Weight合計から来週の負荷を把握

## ステータス判定と対応方法

### 🟢 予定通り（実績 ≥ 予定）

**意味**: 計画通りまたは計画より進んでいます。

**対応**:
- ✅ 現在のペースを維持
- ✅ 余裕があれば次週タスクの準備
- ✅ チーム内で好事例を共有

**注意点**:
- 品質を犠牲にしていないか確認
- バッファを使いすぎていないか確認

### 🟡 やや遅延（予定 - 5% ≤ 実績 < 予定）

**意味**: 若干の遅れがありますが、リカバリー可能です。

**対応**:
1. **原因分析**:
   - どのタスクが遅延しているか確認
   - 遅延の原因を特定（技術的問題、リソース不足等）

2. **リカバリープラン**:
   - 遅延タスクに優先的にリソース配分
   - 次週の予定を見直し
   - 必要に応じてバッファを活用

3. **エスカレーション**:
   - PMに状況を報告
   - サポートが必要か相談

**週次レビューでの確認事項**:
- [ ] 遅延の原因は何か？
- [ ] リカバリーは可能か？
- [ ] 追加リソースが必要か？
- [ ] クリティカルパス上のタスクか？

### 🔴 要注意（実績 < 予定 - 5%）

**意味**: 大きな遅延が発生しています。即座の対策が必要です。

**緊急対応（24時間以内）**:
1. **緊急ミーティング開催**:
   - PM、BI Engineer、ステークホルダーで状況共有
   - 遅延の詳細な原因分析

2. **リカバリープラン策定**:
   - クリティカルパスへの影響評価
   - タスクの優先順位見直し
   - 低優先度タスクの延期検討
   - バッファの投入判断

3. **リソース調整**:
   - 追加メンバーのアサイン検討
   - 外部支援の活用検討
   - スコープ調整の検討

4. **エスカレーション**:
   - 経営層への報告
   - 納期への影響評価
   - リスク管理表の更新

**週次レビューでの確認事項**:
- [ ] 遅延の根本原因は何か？
- [ ] プロジェクト納期への影響は？
- [ ] リカバリープランは実行可能か？
- [ ] スコープ変更が必要か？
- [ ] バッファは残っているか？

## タスクステータスの更新方法

進捗レポートの精度を保つため、タスクのステータスを適切に更新する必要があります。

### tasks.json の更新

#### 1. ファイルの場所
```
/home/sh-usami/project-requirements-system/tasks.json
```

#### 2. ステータスの種類

| ステータス | 意味 | いつ設定するか |
|-----------|------|---------------|
| `pending` | 未着手 | タスク作成時の初期値 |
| `in_progress` | 進行中 | タスクに着手したとき |
| `done` | 完了 | タスクが完全に完了したとき |

#### 3. 更新手順

**ステップ1: tasks.json を開く**
```bash
# エディタで開く
code /home/sh-usami/project-requirements-system/tasks.json
# または
vim /home/sh-usami/project-requirements-system/tasks.json
```

**ステップ2: タスクのステータスを更新**

例: TASK-001 を完了にする場合

変更前:
```json
{
  "id": "TASK-001",
  "title": "既存BigQuery DWHテーブルの整合性確認",
  "status": "in_progress",
  ...
}
```

変更後:
```json
{
  "id": "TASK-001",
  "title": "既存BigQuery DWHテーブルの整合性確認",
  "status": "done",
  ...
}
```

**ステップ3: 変更をコミット＆プッシュ**
```bash
cd /home/sh-usami/project-requirements-system
git add tasks.json
git commit -m "Update TASK-001 status to done"
git push origin main
```

**重要**: GitHubにプッシュすることで、次回のレポート生成時に最新のステータスが反映されます。

### ステータス更新のタイミング

#### タスク開始時（pending → in_progress）
- タスクに着手した当日に更新
- 複数タスクを同時進行する場合も、それぞれ in_progress に設定

#### タスク完了時（in_progress → done）
- 成果物が完成し、レビューが完了したとき
- テストが通ったとき
- ドキュメントも含めて完全に完了したとき

**完了の定義**:
- [ ] 成果物が作成された
- [ ] 必要なテストが完了した
- [ ] ドキュメントが更新された
- [ ] レビューが完了した（必要な場合）

### GitHub Projects との連携

GitHub Projects を使用している場合、Issueのステータスと tasks.json のステータスを同期させてください。

**推奨ワークフロー**:
1. GitHub Issue のステータスを更新（Todo → In Progress → Done）
2. tasks.json のステータスを同期して更新
3. git commit & push

## レポートの活用方法

### 1. 日常的な活用

#### 毎週月曜日の朝（レポート受信時）

**PMの確認事項**:
- [ ] 進捗率のステータスを確認（🟢/🟡/🔴）
- [ ] 🟡または🔴の場合、原因を分析
- [ ] 今週完了タスクが予定通りか確認
- [ ] 来週のタスクの準備状況を確認

**BI Engineerの確認事項**:
- [ ] 自分の担当タスクの進捗を確認
- [ ] 遅延しているタスクがあれば対策を考える
- [ ] 来週のタスクに必要な準備を開始

#### 週次レビューミーティング（毎週月曜 10:00-10:30）

レポートを元に以下を議論:
1. 今週の完了タスクのレビュー
2. 進捗率の確認と遅延原因の分析
3. 来週のタスク確認と準備状況
4. リスク・課題の洗い出し
5. バッファ消費状況の確認

### 2. Phase終了時の活用

#### Phase 1 終了レビュー（Week 4: 1/31）

レポートから確認:
- [ ] Phase 1 の全タスクが完了しているか（Weight 40% 達成）
- [ ] 成果物が全て揃っているか
- [ ] Phase 2 開始の準備ができているか

#### Phase 2 終了レビュー（Week 9: 3/5）

レポートから確認:
- [ ] Phase 2 の全タスクが完了しているか（Weight 71% 達成）
- [ ] Looker Studio ダッシュボード3種完成
- [ ] LookerML基本構造実装完了

#### Phase 3 終了レビュー（Week 12: 3/30）

レポートから確認:
- [ ] 全タスクが完了しているか（Weight 100% 達成）
- [ ] 本番リリース完了
- [ ] ドキュメント・トレーニング完了

### 3. ステークホルダーへの報告

**バックオフィス責任者への報告**:
- 週次レポートを転送またはCCに追加
- 遅延がある場合は別途説明を追加

**経営層への報告**:
- 月次レポートとして、週次レポート4通を要約
- Phase終了時のマイルストーン達成報告

### 4. リスク管理

レポートから以下のリスクを早期検知:

| リスク | 検知方法 | 対策 |
|--------|---------|------|
| 技術的な遅延 | 特定タスクが長期間 in_progress | 技術支援、外部リソース |
| バッファ消費 | 連続して🟡または🔴 | スコープ調整、リソース追加 |
| 依存関係の問題 | 来週タスクの前提が未完了 | タスク順序の見直し |
| クリティカルパス遅延 | クリティカルパス上のタスクが遅延 | 最優先で対処 |

## トラブル時の対応

### レポートが届かない場合

**確認手順**:
1. 迷惑メールフォルダを確認
2. GitHub Actions の実行ログを確認
   - リポジトリ → Actions → "Weekly Progress Report"
3. SendGrid のダッシュボードで送信履歴を確認

**対処方法**:
- `docs/PROGRESS_REPORT_SETUP.md` のトラブルシューティングを参照
- GitHub Actions を手動実行してテスト

### 進捗率が正しくない場合

**原因**:
- tasks.json のステータスが更新されていない
- GitHubへのプッシュが忘れられている

**対処方法**:
1. tasks.json の全タスクのステータスを確認
2. 完了したタスクを `done` に更新
3. git commit & push
4. 次週のレポートで反映される

### 手動でレポートを再生成したい場合

**方法1: GitHub Actionsで手動実行**
1. リポジトリの Actions タブを開く
2. "Weekly Progress Report" を選択
3. "Run workflow" をクリック
4. ブランチを選択して実行

**方法2: ローカルで実行**
```bash
cd /home/sh-usami/project-requirements-system

# 環境変数を設定
export SENDGRID_API_KEY="your-api-key"
export REPORT_TO_EMAIL="your-email@example.com"
export REPORT_FROM_EMAIL="noreply@yourdomain.com"

# スクリプトを実行
python scripts/send-progress-report.py
```

## ベストプラクティス

### 1. タスクステータスの更新タイミング

**推奨**:
- タスク開始時: その日のうちに in_progress に更新
- タスク完了時: 完了した当日に done に更新
- 金曜日の夕方: 週の最後に全体のステータスを見直し

### 2. レポートのアーカイブ

**メールのラベル設定**:
- Gmail: `[プロジェクト名] 週次レポート` ラベルを作成
- Outlook: フォルダを作成して自動振り分け

**重要なレポートの保存**:
- Phase 終了時のレポート
- 🔴 要注意が出たときのレポート
- プロジェクト完了時のレポート

### 3. チーム内の情報共有

**Slackでの共有**:
```
📊 Week N 進捗レポート

✅ 予定通り進行中！
- 予定: 22% → 実績: 25% (+3%)
- 今週完了: TASK-001, TASK-002
- 来週予定: TASK-005, TASK-003, TASK-006

詳細はメールをご確認ください 📧
```

### 4. 定期的な振り返り

**月次振り返り**:
- 4週分のレポートを見返す
- 進捗率の推移をグラフ化
- 遅延の傾向を分析
- プロセス改善を議論

## まとめ

週次進捗レポートを効果的に活用することで:

✅ **透明性**: 全員が同じ情報を持つ
✅ **早期発見**: 遅延を早期に検知できる
✅ **迅速な対応**: 問題に素早く対処できる
✅ **モチベーション**: 進捗の可視化でチームの士気向上
✅ **記録**: プロジェクトの歴史として保存

レポートは**ツール**です。活用方法次第でプロジェクトの成功確率が大きく変わります。
