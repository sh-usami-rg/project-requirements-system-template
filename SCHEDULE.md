## 進捗率計算ルール

### Weight配分
- 総Weight = 100%（プロジェクト全体）
- 各タスクのWeightは工数や重要度に応じて配分
- Done状態になったタスクのWeightを累計して実績進捗率を計算

- **リスク**:
  1. ユーザー要件変更によるスコープクリープ
  2. 50%稼働による遅延リスク
- **対策**:
  1. 優先度による段階的実装
  2. 変更管理プロセスの明確化
  3. クリティカルパスの厳格な管理
  4. 早期のリスク検知とエスカレーション
- **影響Week**: Week 9-12（全タスク）
- **バッファ**: Phase 3全体で約1日

## バッファ管理

### Phase別バッファ
- **Phase 1**: 約1日（Week 1-4）- 設計段階での不確実性吸収
- **Phase 2**: 約1日（Week 5-8）- 技術習熟・実装の不確実性吸収
- **Phase 3**: 約1日（Week 9-12）- テスト・調整の不確実性吸収
- **総バッファ**: 約3日（総工数28.5日の10%相当）

### バッファ消費ルール
1. 遅延が発生した場合、まずPhase内バッファを使用
2. Phase内バッファを使い切った場合、次Phaseと調整
3. 総バッファの50%（1.5日）を超えた場合、エスカレーション
4. クリティカルパス上のタスクが遅延した場合は即座にエスカレーション

### バッファ状況モニタリング
- **週次確認**: 毎週月曜日の進捗会議でバッファ消費状況を確認
- **Phase終了時**: 各Phase終了時に残存バッファを確認し、次Phase計画を調整

## 進捗トラッキング方法

### GitHub Projects設定
1. Project Board作成: "顧問ミドル運用ダッシュボード移行"
2. カスタムフィールド追加:
   - **Status**: Todo, In Progress, Done
   - **Weight**: Number型（0-100）
   - **Phase**: Select型（Phase 1, Phase 2, Phase 3）
   - **Week**: Select型（Week 1-12）
   - **Assignee**: BI Engineer
   - **Priority**: Select型（high, medium, low）

3. タスクをIssueとして登録（25個）
4. Weightを設定（tasks.jsonのWeight配分に従う）
5. 依存関係をIssue間リンクで表現

### 進捗確認方法
```bash
# 現在の進捗を確認（tasks.jsonから計算）
# 完了タスクのWeight合計 / 100 = 実績進捗率
cat tasks.json | jq '.tasks | map(select(.status == "done") | .weight) | add'

# 週次の予定進捗率確認
cat schedule.json | jq '.weeklySchedule[] | {week: .week, progress: .cumulativeProgress}'
```

### 週次レポート
- 毎週金曜日に進捗レポート作成
- 含める内容:
  - 今週完了タスク
  - 実績進捗率 vs 予定進捗率
  - バッファ消費状況
  - 来週の予定タスク
  - リスク・課題
